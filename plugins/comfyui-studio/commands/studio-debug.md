# /studio-debug

诊断 ComfyUI 工作流执行中遇到的问题，输出结构化的诊断报告和修复建议。

## 执行流程

### 1. 加载诊断知识

读取以下参考资料：

- `skills/troubleshooter/SKILL.md` — 故障排查知识库（如存在）
- `skills/model-manager/references/vram-table.md` — VRAM 估算参考

### 2. 读取环境信息

读取 `.comfyui-studio/inventory.json` 获取：
- GPU 型号和显存
- 推荐档位
- 已安装模型和节点清单

如果 inventory.json 不存在，提示用户：

```
未找到环境配置文件。建议先运行 /studio-scan 扫描环境，
这有助于更准确地诊断问题。

是否继续无环境信息的诊断？
```

### 3. 收集错误信息

向用户询问或从上下文中获取错误信息：

```
请提供以下信息（越详细越好）：

1. 错误消息 — 粘贴 ComfyUI 控制台中显示的报错内容
2. 操作描述 — 你在做什么操作时出错的？
3. 工作流信息 — 使用了哪些模型和节点？（如知道）
```

### 4. 启动诊断

启动 **comfy-debugger** agent，将收集到的错误信息和环境配置传入。

comfy-debugger 将：
1. 分析错误类型（OOM / 节点缺失 / 连接失败 / 模型不兼容 / 数值异常 / 其他）
2. 交叉校验 inventory.json 中的环境信息
3. 生成针对性的修复方案

### 5. 输出诊断报告

将 comfy-debugger 的诊断结果以格式化报告输出给用户：

```
========================================
  ComfyUI 工作流诊断报告
========================================

错误类型：{类型}
严重程度：{高/中/低}

错误摘要：
  {一句话概述}

详细分析：
  {分析过程}

修复方案：
  方案 1: {描述}
  方案 2: {描述}
  ...

========================================
```

### 6. 跟进修复

如果用户选择某个修复方案，协助执行：

- **安装节点** → 提供命令并指导安装
- **下载模型** → 调用模型管理功能
- **调整参数** → 修改工作流中的参数设置
- **更换模型** → 推荐替代模型并更新工作流
